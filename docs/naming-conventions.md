# Naming Conventions

## ID Naming Framework
- `id`: Always represents the primary database primary key (CUID generated by Prisma). Use this strictly for relationships, routing targets, and mapping keys (`<div key={item.id}>`).
- `studentId`: Represents the human-readable institutional identification mapping (e.g., 'STU-1002'). Exclusively for display purposes within lists or profiles.
- `teacherId` / `courseId`: Depending on context, inside a relational model (like `Submission`), this defines the foreign key linking to a root `id`. Inside the `Teacher` model itself, `employeeId` represents the human-readable institutional mapping.

## File Naming
- **Components & Layouts:** Strict kebab-case. (e.g., `dashboard-skeleton.tsx`, `teacher-error-boundary.tsx`).
- **Data Hooks & Server Actions:** Kebab-case. (e.g., `get-data.ts`, `auth-actions.ts`).

## Component Naming
- **Definition:** PascalCase exclusively. (e.g., `export function DashboardSkeleton() {...}`).
- **Instance Variables:** camelCase.

## Server Action Naming
- **Definition Base:** Context-specific camelCase actions prepended by verb. (e.g., `addTeacher`, `updateCourseStatus`, `recordPayment`).
- **Execution Aliasing:** When imported into Context stores or master files, server actions MUST be aliased explicitly with a `db` prefix to eliminate shadowing. (e.g., `import { addTeacher as dbAddTeacher } from '@/lib/actions/teachers'`).
